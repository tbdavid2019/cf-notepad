## Context

The edit page stores Markdown in a native `textarea` and renders a separate preview. Existing footer controls are generated by `src/templates/common.js`, while edit-page behavior is initialized in `src/templates/base.js`. The project has no frontend framework or bundler for this feature.

## Goals / Non-Goals

**Goals:**

- Add a compact, icon-first toolbar directly above the editor.
- Make commands work for both selected text and an empty caret position.
- Keep commands deterministic and local so the existing save/render pipeline continues to receive a normal `input` event.
- Support desktop and narrow mobile layouts without hiding essential editor access.

**Non-Goals:**

- Replace the textarea with a WYSIWYG editor.
- Add rich clipboard export or platform-specific publishing.
- Add backend APIs, persistence, or third-party dependencies.

## Decisions

- Use native textarea selection APIs instead of `contenteditable`. This preserves the current Markdown source of truth and avoids coupling the preview renderer to a second editing model.
- Keep command definitions in a small client-side command table. Each command supplies prefix/suffix or block insertion behavior and can be tested independently through source-level tests.
- Render the toolbar in the edit page only. Read-only/share pages retain their current layout.
- Use the existing SVG icon style and toolbar design tokens. On narrow screens, allow horizontal scrolling rather than wrapping the editor content unpredictably.

## Risks / Trade-offs

- [Risk] Markdown block commands can produce surprising results for mixed selections → use conservative line-based transformations and keep the selected range selected after insertion.
- [Risk] Some commands may be unfamiliar → provide localized `title` and `aria-label` text.
- [Risk] Toolbar consumes vertical space on mobile → keep it compact and horizontally scrollable.

## Migration Plan

1. Add pure command behavior tests.
2. Add the command implementation and wire it to the edit template.
3. Add toolbar styles and run the existing test/build checks.

Rollback is additive: remove the toolbar markup, command wiring, and styles; stored Markdown is unchanged.

## Open Questions
